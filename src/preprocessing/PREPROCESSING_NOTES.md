# Preprocessing Approach — Comparison with Sai et al. (2023)

## What the Reference Paper Did (Section 3.1)

Sai et al. operated on documents obtained as PDFs and described preprocessing as
"semi-automated with multiple cyber-human interaction options":

| Challenge | Their Solution |
|-----------|----------------|
| **PDF → plain text** | Automated extraction, but **manual review strongly encouraged** |
| **Sentence boundary detection** | Manual review required — PDFs produce broken lines, hyphenation artifacts, page-break noise |
| **Anaphora resolution** | Automated tool to make each sentence self-contained ("it", "they" → explicit referent) |
| **Relevance filtering** | Signal words ("shall", "should", "must") identify constraint sentences |
| **Responsibility mapping** | Semi-automated entity extraction for responsibility terms |

Their best retrieval result (Legal-S-BERT enhanced) achieved **70.6% recall**,
with overall deviation detection precision around **55%** — with manual review
as a prerequisite.

## Why Most of These Challenges Do NOT Apply to Us

### GDPR (regulatory document)

We use `data/gdpr/L_2016119EN.01000101.html` — the **official EUR-Lex XHTML**
published directly by the EU (Official Journal L 119, 4 May 2016, converted to
XHTML on 2024-12-11 by EUR-Lex CONVEX v9.16.1).

This eliminates the paper's primary preprocessing burden entirely:

| Paper's PDF challenge | Our situation |
|----------------------|---------------|
| OCR artifacts / encoding errors | No OCR — source is machine-generated XML/XHTML |
| Hyphenation across line breaks | Not present — HTML text nodes are complete words |
| Page headers/footers bleeding into text | Completely absent from article `<div>` nodes |
| Multi-column layout confusion | Table-based layout is semantically tagged; handled by `extract_gdpr.py` |
| Sentence boundary detection | Each `<p>` tag is one logical sentence/clause — boundaries are explicit |
| **Manual review needed** | **Not needed — text quality is publication-grade** |

Articles 5–43 are individually addressable via `id="art_N"` anchors. List items
(sub-clauses (a), (b), …) use a consistent two-column HTML table structure
(4% marker / 96% content) which `extract_gdpr.py` reconstructs cleanly.

**Result:** `src/preprocessing/extract_gdpr.py` produces `data/gdpr/gdpr_art5_43.txt`
in one automated pass with no manual review step. 85,547 characters across 441
logical text lines, zero noise.

### Internal Policy (realization)

`data/policy/novatech_data_protection_policy.txt` was **generated by us** as clean
plain text. There is no PDF-to-text conversion involved at all.

## What Still Applies

### Anaphora / cross-reference resolution (partially)

GDPR text contains cross-article references ("as referred to in Article 89(1)")
and within-article pronoun references ("it shall", "they may"). These require
awareness of context when a constraint sentence is extracted in isolation.

**Mitigation:** For the constraint extraction step, we include enough context
(full paragraph, not just the triggering signal-word sentence) so that the LLM
classifier in Step 3 can resolve local pronouns without explicit anaphora
resolution tooling. This is a deliberate simplification relative to the paper,
enabled by using an LLM rather than a rule-based classifier.

### Relevance filtering

Signal words ("shall", "should", "must") are still needed to identify constraint
sentences from the full article text. This is identical to the paper's approach.
See `extract_constraints.py` (to be built) for implementation.

## Practical Advantage

Because our input sources are structurally clean, we can invest preprocessing
effort entirely in **semantic** tasks (constraint extraction, decomposition)
rather than **format** tasks (cleaning, boundary detection, manual review).
This is a direct improvement over the paper's baseline setup and a legitimate
methodological contribution to note in the report (Section 4 / Implementation).
